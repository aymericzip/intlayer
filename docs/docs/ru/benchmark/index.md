# Benchmark Bloom — Отчет

Benchmark Bloom — это набор тестов производительности, который измеряет реальное влияние библиотек i18n (интернационализации) на различные React-фреймворки и стратегии загрузки.

Подробные отчеты и техническую документацию для каждого фреймворка можно найти ниже:

- [**Отчет о бенчмарке Next.js**](https://github.com/aymericzip/intlayer/blob/main/docs/docs/ru/benchmark/nextjs.md)
- [**Отчет о бенчмарке TanStack Start**](https://github.com/aymericzip/intlayer/blob/main/docs/docs/ru/benchmark/tanstack.md)

## Текущие результаты

Посетите [**интерактивную панель мониторинга бенчмарка**](https://intlayer.org/benchmark) для просмотра живых сравнений и обобщенных данных.
| `scoped-dynamic` | Высокая (утечка близка к нулю) | Высокая |

Переход от `static` к `scoped-dynamic` обычно сокращает неиспользуемый контент на 60–90%, но требует значительно больше конфигурации. Библиотеки, такие как Intlayer, автоматизируют паттерн `scoped-dynamic`, поэтому разработчики получают эффективность без лишнего шаблонного кода.

### Чтение показателей утечки

Утечка страницы в **35%** означает, что 35% JavaScript, загруженного для этой страницы, содержит строки с других страниц — контент, который пользователь не может видеть на этой странице. На странице размером 400 КБ это составляет ~140 КБ данных, которых можно было бы избежать.

Утечка локали в **10%** означает, что 10% бандла содержит переводы на языки, которые текущий пользователь не использует.

### Реактивность против времени рендеринга

- **E2E реактивность** измеряет полный пользовательский опыт: сеть, накладные расходы фреймворка, обновление DOM.
- **Время React Profiler** изолирует стоимость перерендеринга дерева React.

Библиотека может иметь низкое время Profiler, но высокое время E2E, если переключение локали включает сетевой запрос (получение нового файла локали). И наоборот, библиотека может иметь высокое время Profiler, но при этом ощущаться быстрой, если она эффективно группирует обновления.
